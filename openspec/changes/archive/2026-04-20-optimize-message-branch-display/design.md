## Context

revision 表格当前以列布局展示 Change ID、Message、Author、Date 等信息。分支名位于顶部区域时，与具体 revision 的 message 关联不够直接，也会占用表格上方空间；完整 message 多行展示则可能破坏固定行高和列表扫读效率。

## Goals / Non-Goals

**Goals:**
- 将分支名从顶部区域移动到对应 revision 的 Message 单元格左侧
- Message 单元格只展示 message 第一行，避免多行正文撑高行
- 保持现有列宽、固定行高、虚拟滚动和横向滚动对齐行为

**Non-Goals:**
- 不改变底层 revision 数据模型或 jj 数据获取逻辑
- 不新增分支筛选、分支排序或分支管理能力
- 不调整 Author、Date、Change ID 等其他列的展示规则

## Decisions

1. 分支名作为 Message 单元格内的前缀展示。
   - 理由：分支名与提交说明关系最直接，放在同一单元格可以减少顶部信息区占用，并避免新增列导致横向空间进一步紧张。
   - 备选方案：新增 Branch 列。该方案会增加列宽配置和横向滚动压力，且用户请求明确要求移动到 message 左侧。

2. Message 展示层提取第一行，不改变原始数据。
   - 理由：保留完整 message 供后续详情视图或其他功能使用，列表层只负责紧凑展示。
   - 备选方案：在数据加载时截断 message。该方案会丢失上下文，不利于后续功能复用。

3. 顶部区域不再显示分支名。
   - 理由：分支名已经与行内 revision 绑定展示，顶部重复显示会制造视觉噪音。
   - 备选方案：顶部和行内同时显示。该方案信息重复，且不符合“取消顶部的分支名显示”的要求。

## Risks / Trade-offs

- 分支名较长可能挤压 message 第一行可见空间 → 使用固定最大宽度、截断和 tooltip/title 等方式保留可识别性
- 多个分支名关联同一 revision 时可能造成单元格拥挤 → 以紧凑 tag 形式展示，并限制整体宽度
- 只显示 message 第一行会隐藏正文细节 → 仅影响列表展示，原始完整 message 数据保持不变
