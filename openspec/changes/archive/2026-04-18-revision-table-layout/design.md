## Context

目前的提交图渲染依赖于传统的节点连线方式，在分支较多或历史复杂的仓库中，视觉上容易产生混叠。为了提高可读性，我们需要引入一种“表格化”布局（Grid-based Table Layout），通过明确的水平（时间轴）和垂直（分支/逻辑轴）分割线来组织修订版本。

## Goals / Non-Goals

**Goals:**
- 实现基于网格（Grid-based）的提交历史展示，**取代现有的 Canvas 渲染逻辑**。
- 每个修订版本（Revision）占据一个独立的网格单元，实现结构化的表格布局。
- 使用浅色分割线明确划分行（修订版本）和列（稳定的修订轨道）。
- 确保所有现有的交互功能（选择、重定基底/Rebase、右键菜单、折叠/展开）在新的网格布局中完整保留并得到优化。
- 提升在复杂分支结构下的视觉清晰度，确保主干始终稳定。

**Non-Goals:**
- 保留旧的 Canvas 渲染代码（一旦新布局稳定，旧代码将被移除或弃用）。
- 改变 `jj` 核心的数据结构。

## Decisions

### 1. 布局引擎选择：CSS Grid (Hard-Locked)
**决策：** 使用 `CSS Grid` 并配合**严格的像素锁定**（Fixed Pixel Sizes）。
**理由：**
- `CSS Grid` 提供了现代的网格管理能力。
- 为了确保背景 SVG 连线与单元格 100% 对齐，我们放弃了“弹性布局”，改用硬性的 `columnWidth` 和 `rowHeight` 像素值计算。
- 强制设置 `whitespace-nowrap` 和 `overflow-hidden`，防止单元格内容撑开网格导致形变。

### 2. 身份识别基准：ChangeId
**决策：** 使用 `change_id` 而非 `commit_id` 作为布局计算的唯一 Key。
**理由：**
- 在 `jj` 中，`change_id` 是修订版的稳定身份。使用它作为 Key 可以确保在执行 `describe`、`amend` 等操作导致 `commit_id` 变化时，修订版在网格中的列位置保持绝对稳定。

### 3. 核心算法：自底向上预留轨道算法 (Bottom-Up Gap-Preserving)
**决策：** 逆向遍历时间线（从老到新）进行列分配，并实施全局轨道锁定。
**理由：**
- **垂直轴**：严格对应 `jj log` 索引，不做压缩，完整保留时间感。
- **水平轴**：最高优先级的子节点（如 `main` 或 `@` 所在路径）继承父节点列号。
- **预留间隙**：次级优先级分支在右侧寻找“全局从未被占用”的空列，且该列在分支生命周期内不被其他无关提交顶替。

### 4. 层级对齐架构
**决策：** 将 Grid 坐标样式提升到容器的直接子元素（Direct Children），并在此层级包裹 `CommitContextMenu`。
**理由：**
- 修复了因为中间组件（ContextMenu）阻断 CSS Grid 坐标继承导致的单元格“被挤走”问题。
- SVG 连线层置于单元格下方（`zIndex: 5`），单元格背景保持透明。

## Risks / Trade-offs

- **[Risk] 空间利用率**：强制保留间隙会导致横向空间占用较大。
  - **Mitigation**：增加横向滚动支持，并允许用户调整 `columnWidth`。
- **[Risk] 渲染性能**：大量 DOM 节点。
  - **Mitigation**：后续引入虚拟滚动（Task 4.3）。
