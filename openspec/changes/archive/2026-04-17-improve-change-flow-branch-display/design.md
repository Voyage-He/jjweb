## Context

当前变更流程图（Change Flow Diagram）在展示分支结构时存在局限。当多个子变更从同一个父变更分出时，它们以线性方式堆叠显示，无法清晰体现并行开发的关系。用户在查看变更流程时难以分辨哪些变更是并行开发的分支关系，哪些是串行的依赖关系。

本项目需要扩展 `commit-graph` 的布局算法，并引入 `branch-visibility` 能力来实现分支结构的准确展示。

## Goals / Non-Goals

**Goals:**
- 实现同一父变更的多个子变更以水平分支形式展示
- 为每个分支提供视觉区分（颜色、线条样式）
- 清晰标识分支点（branch point）和合并点（merge point）
- 支持分支的折叠/展开功能
- 保持与现有提交图布局的向后兼容

**Non-Goals:**
- 不修改 jj (Jujutsu) 本身的存储或变更数据模型
- 不实现跨仓库的分支展示
- 不实现自动合并冲突的可视化

## Decisions

### Decision 1: 分支布局算法采用水平分层布局（H-Tree Layout）

**选择:** 使用改进的 H-Tree 布局算法

**理由:**
- H-Tree 天然支持从父节点水平分出多个子节点，非常适合展示分支结构
- 布局算法复杂度为 O(n)，适合大规模变更图的渲染
- 与现有的纵向下钻布局（vertical drill-down）正交，可以共存

**替代方案考虑:**
- **Dagre/Dagre.js**: 图布局库，适合有向图的自动布局。但引入额外依赖，且对分支可视化支持不够直观
- **Force-directed layout**: 适合展示关系网络，但结果不稳定，难以预测节点位置，不利于用户导航

---

### Decision 2: 分支视觉区分采用颜色 + 线型组合

**选择:** 每个分支分配独立颜色 + 不同的连接线型（实线/虚线）

**理由:**
- 颜色是最直观的视觉区分方式，符合用户直觉
- 线型作为辅助区分，在黑白打印或色盲用户场景下仍可辨识
- 颜色方案采用 HSL 色彩空间的色相轮转，确保相邻分支颜色对比度足够

**替代方案考虑:**
- **仅使用颜色**: 在色盲用户中辨识度低
- **仅使用线型**: 分支多时难以区分

---

### Decision 3: 分支数据结构采用邻接表 + 深度信息

**选择:** 使用邻接表（Adjacency List）存储分支关系，每个节点记录 `branchDepth` 和 `siblingIndex`

**理由:**
- 邻接表查询效率高，适合频繁查询父子关系
- `branchDepth` 用于计算水平偏移量
- `siblingIndex` 用于计算垂直对齐位置
- 数据结构轻量，易于 JSON 序列化

**替代方案考虑:**
- **嵌套树结构**: 适合显示但修改困难（特别是 rebase 操作后）
- **边列表**: 查询需要遍历，适合静态图但不适合动态更新

---

### Decision 4: 折叠/展开状态由前端组件本地管理

**选择:** 折叠状态存储在 React/Vue 组件 state 中，不持久化到全局状态

**理由:**
- 折叠是临时性的视图偏好，不应影响变更数据本身
- 本地状态响应速度快，用户体验流畅
- 实现简单，无需考虑状态同步问题

**替代方案考虑:**
- **持久化到 localStorage**: 用户切换设备后期望保持折叠状态，但增加了复杂度
- **全局状态管理（Redux/Vuex）**: 对于折叠状态过于重量级

---

### Decision 5: 分支合并点的检测在渲染前完成

**选择:** 在图渲染阶段之前，遍历变更图检测合并点并标记

**理由:**
- 合并点是图的拓扑特征，需要全局视图才能准确判断
- 提前计算好合并信息，渲染时直接使用，避免重复计算
- 合并点标记可以作为节点元数据的一部分传递

**替代方案考虑:**
- **实时检测**: 用户拖拽或缩放时实时检测，性能开销大
- **Web Worker 后台计算**: 增加架构复杂度，收益不明显

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| 分支过多时水平布局占用屏幕宽度过大 | 设置最大分支宽度，超出后自动换行或缩放 |
| 颜色方案在高对比度模式下不可见 | 同时保留线型区分作为 fallback |
| 折叠状态切换时布局跳动明显 | 添加 CSS transition 动画，平滑过渡 |
| 分支合并点检测在大规模图上性能下降 | 采用虚拟化渲染，仅绘制可视区域节点 |

## Open Questions

1. **分支颜色方案的数量上限**: 最多支持多少个同时显示的分支而不造成颜色混淆？
2. **折叠节点的默认展开层级**: 用户进入视图时，分支默认是展开还是折叠状态？
3. **移动端适配**: 水平分支布局在小屏幕设备上是否需要切换为垂直堆叠？
