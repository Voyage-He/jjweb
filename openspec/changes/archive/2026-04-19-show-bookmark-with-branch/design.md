## Context

RevisionTable 组件使用 CSS Grid 布局将修订版本渲染为网格，每个修订版本占据一个单元格。当前布局：

- 使用 `gridTemplateColumns` 定义列宽
- 使用 `gridAutoRows` 定义行高
- 每个单元格通过 `gridRow` 和 `gridColumn` 定位

约束：
- 需要保持虚拟滚动功能
- 需要与现有网格布局兼容
- 书签数据已存在于 `Commit.bookmarks` 数组中

## Goals / Non-Goals

**Goals:**
- 在每列顶部添加一个固定区域显示书签名称
- 自动检测该列中任意单元格的书签并显示
- 无书签时显示空白区域
- 悬停时显示该列所有书签的完整列表
- 保持现有网格布局的稳定性和性能

**Non-Goals:**
- 不修改 CommitGraph（Canvas 渲染）组件
- 不修改书签的数据结构或 API
- 不添加列头的复杂交互功能（如点击排序、右键菜单）

## Decisions

### 1. 列头实现方式

**选择：在 Grid 外部添加独立的列头行**

在网格上方添加一个独立的 flex 容器，与网格列宽对齐。

**理由：**
- 不需要修改现有网格结构
- 始终可见（不会随滚动消失）
- 实现简单，性能开销小

**替代方案：**
- 使用 sticky 定位在网格内部渲染 → 会增加网格复杂度，可能影响虚拟滚动
- 使用 frozen row → CSS Grid 不原生支持

### 2. 书签检测策略

**选择：预处理所有 commits，按列分组提取书签**

遍历所有 commits，按 `column` 属性分组，提取每列的书签名称。

**理由：**
- 计算简单，O(n) 复杂度
- 利用现有的 `Commit.column` 属性
- 可通过 `useMemo` 缓存结果

### 3. 书签显示规则

**选择：每列显示该列第一个（顶部）具有书签的修订版本的书签**

按 `row` 从小到大遍历，取第一个有书签的修订版本的书签名称。

**理由：**
- 符合用户预期（书签通常指向分支的最新提交）
- 避免一列显示多个书签导致的视觉混乱
- 与 Git 分支显示习惯一致

### 4. 多书签悬停交互

**选择：使用原生 HTML title 属性显示完整书签列表**

当鼠标悬停时，通过 `title` 属性显示所有书签名称。

**理由：**
- 实现简单，无需额外组件
- 浏览器原生支持，性能好
- 各平台一致的 UX 体验

**替代方案：**
- 自定义 Tooltip 组件 → 增加实现复杂度，对于简单文本显示不必要

## Risks / Trade-offs

- **书签移动延迟**：书签移动后需等待刷新 → 接受，与现有行为一致
- **title 属性延迟**：浏览器 tooltip 显示有短暂延迟 → 接受，符合用户预期
